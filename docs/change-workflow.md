# Change Workflow

Use small phases. A good phase can be reviewed, tested, committed, and rolled back without understanding the whole app.

## Before Any Change

1. Check `git status --short`.
2. Inspect related routes, controller, model, view, JavaScript, migration, and tests.
3. Identify data/runtime impact.
4. Identify permissions and feature flags.
5. List exact files to change if approval is required.

## Adding A New Feature

Checklist:

- define module or existing module ownership
- add route names and controller actions
- add validation
- add role checks
- add feature flag if it can be client-disabled
- add labels if wording may vary by client
- add views/components
- add page JavaScript if needed
- add tests
- update release/package rules only if new source paths are introduced

## Adding A Client-Specific Change

First decide what kind of difference it is:

- wording difference: label config
- access difference: feature flag
- process difference: workflow config
- package ownership difference: module manifest/pruning
- truly unique UI/code: overlay as last resort

Example:

Client A wants Article renamed to Design and Shipments disabled. Use label config for Article -> Design and feature/module config for Shipments. Do not fork the whole invoice module.

## Adding A Global Bug Fix

Global bug fixes should reach every client package unless a module is pruned.

Example:

Invoice Quick Print is a global invoice fix. Client A should still receive it even if Article is labeled Design and Shipments are disabled.

## Route Cleanup Workflow

When a resource controller has blank methods:

1. find all routes generated by `Route::resource`
2. confirm which actions are used by views/JS
3. replace with `only([...])` or explicit routes
4. add tests for unsupported direct URLs if risk is high
5. do not change business behavior

## Testing

Use focused tests first:

- affected feature tests
- affected unit tests
- route/permission tests
- service tests

Run full suite when the change affects shared middleware, auth, route behavior, reports, release scripts, backup, or runtime safety.

Docs-only changes usually need:

```text
git diff --check
```

## Commit Workflow

- One phase per commit.
- Keep unrelated changes out.
- Mention tests in the commit body.
- Never commit `.env`, database files, uploads, logs, backups, or runtime folders.
- Check `git status --short` before and after.

## Release Rule Updates

Update release rules only when source/package ownership changes. Do not loosen exclusions to make a package pass. Investigate why the file is present.
